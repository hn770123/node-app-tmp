# AWS App Runner 導入・コスト試算およびセキュリティ評価レポート

## 1. コスト試算と6ヶ月間の試用判定

AWS App RunnerはAWSの無料利用枠（Free Tier）の対象外ですが、最小構成（1 vCPU / 2GB RAM）を利用し、テスト時のみアクティブにしてそれ以外は一時停止（Pause）する運用を前提としてコストを試算します。

### 前提条件
*   **コンテナ設定**: 1 vCPU, 2GB RAM
*   **利用リージョン**: 東京リージョン (ap-northeast-1) を想定
*   **運用方法**: 必要なときだけ稼働（アクティブ）、不要なときは「一時停止（Pause）」
*   **想定稼働時間**: テスト目的のため、月間アクティブ稼働時間は **10時間**、一時停止状態でプロビジョニングされている時間を **0時間**（一時停止中は課金されません）と仮定します。
    *   *※App Runnerは「アクティブ状態」と「プロビジョニング（待機）状態」で課金が変わります。一時停止状態（Pause）にするとコンテナのプロビジョニングが解除され、コンピューティング・メモリの両方の課金が停止します。*

### 単価設定 (東京リージョンの場合)
*   **アクティブコンテナインスタンスのコンピューティング**: $0.081 / vCPU-hour
*   **プロビジョニングされたコンテナインスタンスのメモリ**: $0.009 / GB-hour

### 月額コスト試算
月間10時間だけ稼働（Resume -> Active）し、残りの時間は一時停止（Pause）する場合：
*   **コンピューティングコスト**: 1 vCPU × 10時間 × $0.081 = $0.81
*   **メモリコスト**: 2 GB × 10時間 × $0.009 = $0.18
*   **月額合計コスト**: **$0.99**

### 6ヶ月間の試用判定
*   **6ヶ月間の合計コスト**: $0.99 × 6ヶ月 = **$5.94**
*   **判定結果**: ユーザーが希望する「月額 $10 程度」という予算に対して、**十分に収まります**。
    *   *ただし、一時停止（Pause）を忘れてプロビジョニング状態（待機状態）のまま1ヶ月（730時間）放置した場合、メモリコストとして 2 GB × 730時間 × $0.009 = $13.14 /月 が発生し、$10の予算を超過するため、こまめなPause操作が必須です。*

## 2. セキュリティおよび運用上の問題点

AWS App Runnerへのデプロイにおいて、現在のプロジェクト構成 (`app.js` を中心とする) をそのまま移行した場合に懸念される、セキュリティ面や運用上の課題を指摘します。
※ 今回は「移行不要（現在のSQLite構成のまま）」との要件に基づき、修正の提案ではなく課題の指摘に留めます。

### 1. データ永続性の欠如 (コンテナの揮発性)
**【課題】**
App Runnerのコンテナインスタンス上のファイルシステムは「エフェメラル（揮発性）ストレージ」です。再起動、スケールアウト、そして今回想定されている「**一時停止（Pause）**」の操作を行うと、コンテナが破棄され、ローカルに保存されているすべてのファイルが消失します。
**【影響】**
現在のコードは以下のデータをローカルファイルに保存しています。
*   `database.sqlite` (ユーザー情報やデータ)
*   `sessions.sqlite` (ログインセッション情報)
これらのファイルが再起動や一時停止のたびに初期化されるため、ユーザーの作成状態やログイン状態が都度リセットされ、実質的に運用が困難になります。

### 2. 同期処理 (Sync) によるイベントループのブロッキング
**【課題】**
`app.js` のログインやユーザー作成時に、パスワードのハッシュ化・照合に `bcrypt` ライブラリの同期メソッドが使用されています。
*   `bcrypt.compareSync(password, user.password)`
*   `bcrypt.genSaltSync(10)`
*   `bcrypt.hashSync(password, salt)`
**【影響】**
Node.jsはシングルスレッドで動作するため、重いハッシュ計算を同期処理で行うと、その処理中（数十〜数百ミリ秒）は他のすべてのリクエスト処理が停止（ブロック）されます。これにより、意図的な大量のログイン試行（DoS攻撃）を受けた際にアプリケーション全体が応答不能になるセキュリティリスクがあります。

### 3. セッション管理のハードコードされたシークレットキー
**【課題】**
`app.js` 内のセッション設定において、環境変数 `SESSION_SECRET` が指定されていない場合のフォールバック値がハードコードされています。
```javascript
secret: process.env.SESSION_SECRET || 'super_secret_key_change_in_production'
```
**【影響】**
本番環境（App Runner環境）で適切に環境変数が設定されずに起動された場合、推測可能なデフォルトのシークレットキーが使用されてしまい、セッションハイジャックなどの重大なセキュリティリスクにつながります。

### 4. 未捕捉例外によるサーバー停止の可能性
**【課題】**
非同期のデータベース操作 (`db.get` や `db.run` など) でコールバック内で発生した予期せぬエラーや、同期処理内でキャッチされなかった例外があると、プロセスがクラッシュする可能性があります。
**【影響】**
`uncaughtException` イベントでログ出力の記述はありますが、安全にプロセスを終了・再起動する仕組みが不十分です。App Runnerのヘルスチェックによって再起動は行われますが、サービスの一時的なダウンタイムが発生します。

### 5. `NODE_ENV` の設定漏れによるクッキーのセキュア属性の無効化
**【課題】**
セッションクッキーの設定で、`secure: process.env.NODE_ENV === 'production'` となっています。
**【影響】**
App Runnerデプロイ時に環境変数 `NODE_ENV` を `production` に設定し忘れると、HTTPS通信であってもセキュア属性（暗号化通信でのみクッキーを送信する設定）が付与されず、中間者攻撃に対して脆弱になります。App RunnerはデフォルトでHTTPSエンドポイントを提供するため、この設定は必須です。
